# Project Background — What Is This and Why Do We Need You?
## CS570 Team 14 | For annotators with no prior context

---

## The one-sentence version

We are training an AI model that looks at Android app screens and predicts which text on the screen is sensitive (like prices and passwords) vs. which text can be freely changed (like a promo banner).

---

## The problem we're solving

Imagine you want to automatically modify an Android app's UI — maybe to translate it, personalize it, or A/B test different copy. Not all text is equal:

- The price `$24.99` on a checkout screen **must not be changed** — altering it could deceive users financially.
- The button label `"Add to Cart"` **can be reworded** — it could say "Put in Bag" and that's fine.
- The banner `"Summer Sale — 40% OFF!"` **can be swapped freely** — it's promotional content.

The goal of our project is to build a model that learns to make these distinctions automatically, given only the app screen and its UI structure.

---

## The dataset: Rico

We are using **Rico**, a public dataset of ~72,000 real Android app screenshots collected by researchers at Carnegie Mellon. Each screen comes with:

1. **A PNG screenshot** — what the phone screen looks like
2. **A UI hierarchy JSON** — a tree of every element on that screen (buttons, text fields, images, etc.), with metadata like element type, text content, position on screen, and a unique ID

Think of the hierarchy like the HTML of a webpage — it describes every element and how they're nested inside each other.

---

## What our model does

We represent each screen as a **graph**, where:
- Every UI element (button, text field, image, etc.) is a **node**
- Edges connect parent elements to their children, and siblings to each other

We then train a **Graph Convolutional Network (GCN)** — a type of neural network that operates on graphs — to classify each node into one of three categories:

| Label | What it means |
|-------|---------------|
| **Canonical (0)** | Critical content — must not be changed (prices, account numbers, payment buttons, credentials) |
| **Translatable (1)** | Meaningful text — can be reworded but meaning must be preserved (navigation labels, instructions, field labels) |
| **Open (2)** | Freely replaceable — decorative or promotional content (product photos, banners, taglines) |

The hypothesis is that **graph structure helps** — knowing that a `$24.99` text node is inside a checkout container makes it Canonical, whereas the same `$24.99` inside a promotional banner is Open. A plain model that ignores structure (MLP) can't make that distinction; our GCN can.

---

## How we train it

We use a two-stage approach:

**Stage 1 — Heuristic training labels (automated, large scale)**
We wrote rule-based code that automatically assigns labels to ~5,000–10,000 Rico screens based on patterns in element types, resource IDs, and positions. These are noisy but scale without human effort. We use them only for training and validation.

**Stage 2 — Gold labels (hand-labeled, small scale — this is where you come in)**
For the final test, we need ~50 screens labeled by humans. These gold labels are the ground truth we actually trust. The model is never trained on them — they are used only for the final evaluation, like a hidden exam.

Your annotations are the exam. They determine whether our model actually learned anything real.

---

## Why we can't just use the automated labels for testing too

Because the automated rules and the model are trained on the same assumptions. Testing the model on labels generated by the same rules would be circular — it would look good even if the model learned nothing meaningful. Human labels provide an independent check.

---

## What you're doing

You're a human annotator for **Stage 2**. You will look at ~30 real Android app screenshots (10 shared with the other annotator, 20 only yours), identify each labeled UI element, and assign one of the three labels above.

You do not need to write any code. You just need to:
1. Look at a screenshot
2. Read about each element (text, type, position)
3. Decide: Canonical, Translatable, or Open
4. Fill in a CSV row

The full instructions are in `annotation_guidelines.md`. The quick cheat sheet is in `quick_reference.md`.

---

## Why 10 screens are shared between annotators

Labeling is somewhat subjective, and we need to know how consistent humans are with each other. The 10 overlap screens let us compute **Cohen's kappa** — a standard measure of inter-annotator agreement. If two humans agree most of the time, the labels are reliable. If they frequently disagree, we need to revisit the guidelines.

After both of you submit, we compare your labels on those 10 screens, identify disagreements, and resolve them together before merging into the final gold test set.

---

## The end goal

Once we have your gold labels, we run the trained models on those 50 screens and compare:
- **GCN** (graph-aware) vs. **MLP** (graph-blind)
- The primary claim: GCN should outperform MLP on Macro-F1 because graph structure provides context that plain per-node features cannot

Your labels are the final arbiter of that claim.
