# Implementation Plan - GCN-Based Modifiability Prediction on UI Graphs
**CS570 Team 14 | For coding agent use**

---

## Project Overview

**Task:** Per-node multi-class classification on UI hierarchy graphs.
**Input:** A UI screen rendered as a hierarchy graph `G = (V, E)` with per-node features `x_v`.
**Output:** Label `y_v in {0=canonical, 1=translatable, 2=open}` for every labeled node `v`.
**Dataset:** Rico Android UI hierarchies. Heuristic labels are used for train/validation only; hand-labeled gold labels are the authoritative final test set.
**Headline result:** The GCN-vs-MLP Macro-F1 gap on gold labels, with safeguards showing the result is not just circularly learned from ancestor-based heuristic rules.
**Models:** MLP baseline, GCN, and GAT implemented in PyTorch without PyTorch Geometric unless the course explicitly permits PyG later.
**Environment:** Python 3.10+, PyTorch with CUDA on the GPU server.

## Non-Negotiable Experimental Protocol

1. **Heuristic-labeled Rico screens are for training and validation only.**
   - Default scale: 5K-10K Rico screens before attempting the full 72K dataset.
   - Labels are generated by `src/data/labeler.py`.
   - Early stopping and hyperparameter tuning use only the heuristic validation split.

2. **The final test set is hand-labeled gold data.**
   - Store all gold annotation files under `data/gold/`.
   - Target approximately 50 Rico screens and approximately 2K nodes.
   - Gold screens must come from apps excluded from heuristic train/validation.
   - Gold labels are never used for training, early stopping, class weights, threshold selection, or model selection.

3. **Final evaluation happens once per completed run on `gold_test_labels`.**
   - Report Macro-F1, per-class precision/recall/F1, accuracy, and a 3x3 confusion matrix.
   - The primary comparison is GCN 2-layer all-features containment+sibling vs MLP all-features.

4. **Circularity safeguard is required.**
   - Implement two heuristic label modes:
     - `contextual`: may use ancestor context.
     - `local_only`: may use only node-local text, bounds, class/type, resource-id, and content-desc.
   - Run MLP and GCN trained on `local_only` heuristic labels and evaluate both on gold labels.
   - Report whether the GCN gain persists without ancestor-derived heuristic labels.

### 1.2 Rico Loader (`src/data/rico_loader.py`)

Public interfaces:

```python
from dataclasses import dataclass

@dataclass
class FlattenedHierarchy:
    nodes: list[dict]
    containment_edges: list[tuple[int, int]]
    sibling_edges: list[tuple[int, int]]
    parent_index: list[int | None]
    ancestor_indices: list[list[int]]

def load_hierarchy(json_path: str) -> dict:
    """Load raw Rico JSON and return the root node dict."""

def flatten_hierarchy(root: dict) -> FlattenedHierarchy:
    """
    Flatten a Rico hierarchy into stable node records and graph structure.

    Returns:
      - nodes: node attribute dicts, one per UI element
      - containment_edges: raw directed (parent_idx, child_idx) pairs
      - sibling_edges: raw sibling pairs (child_i_idx, child_j_idx), one unordered pair
      - parent_index: parent index for each node, None for root
      - ancestor_indices: root-to-parent index chain for each node
    """

def get_app_id(json_path: str) -> str:
    """Extract the app/package ID from a Rico JSON path."""
```

Implementation requirements:

- Assign each node a stable per-screen `node_id` and store it inside `nodes[i]["node_id"]`.
- Preserve traversal order deterministically with DFS pre-order or BFS; document which one is used.
- Store `depth`, `child_count`, and `sibling_count` on each node.
- Do not make sibling edges bidirectional in `flatten_hierarchy`; keep them as raw unordered pairs for debugging.
- Preserve enough context for labeling and error analysis through `parent_index` and `ancestor_indices`.

### 1.3 Feature Extraction (`src/data/features.py`)

Each node gets feature groups:

| Group | Contents |
|-------|----------|
| `visual` | normalized bounds, area, aspect ratio, center, relative position within parent |
| `structural` | depth, sibling count, child count, is_leaf |
| `type` | fixed one-hot widget class vocabulary |
| `text` | MiniLM embedding of text/content-desc/resource descriptors |

Visual feature requirements:

- Normalize bounds by actual screen dimensions when available in the Rico hierarchy or metadata.
- Use `1080x1920` only as fallback.
- Clamp invalid widths/heights to non-negative values.
- If parent bounds are missing, parent-relative features are zeros.

Text feature requirements:

- Embed the concatenation of non-empty `text` and `content-desc`.
- Include resource/content descriptors in local feature construction where useful, but do not leak gold labels.
- Empty text/content-desc must produce a zero vector.
- Cache embeddings by normalized text string. Make cache path configurable.

Public interface:

```python
def extract_features(
    flattened: FlattenedHierarchy,
    feature_groups: list[str] | None = None,
    screen_width: int | None = None,
    screen_height: int | None = None,
    embedding_cache_path: str | None = None,
) -> tuple[torch.FloatTensor, dict[str, tuple[int, int]]]:
    """
    Returns:
      features: [num_nodes, dim] float32 tensor
      feature_slices: mapping like {"visual": (0, 12), "structural": (12, 16), ...}
    """
```

Preserve `feature_slices` in every saved graph so ablations can drop groups without recomputing graphs.

Default full dimension is expected to be `12 + 4 + 20 + 384 = 420`, unless the implementation explicitly changes the fixed type vocabulary.

### 1.4 Heuristic Labeling (`src/data/labeler.py`)

Public interfaces:

```python
from typing import Literal

def assign_label(
    node: dict,
    ancestors: list[dict],
    mode: Literal["contextual", "local_only"],
) -> int | None:
    """
    Return 0=canonical, 1=translatable, 2=open, or None for nodes excluded from heuristic training.
    """

def label_graph(
    flattened: FlattenedHierarchy,
    mode: Literal["contextual", "local_only"],
) -> list[int]:
    """Return one label per node, using -1 for unlabeled/excluded nodes."""
```

Label definitions:

- `0 = canonical`: must not be altered; identity, payment, account, or transaction critical.
- `1 = translatable`: text can change form but should preserve meaning.
- `2 = open`: freely modifiable or low-risk creative content.
- `-1`: excluded from loss/evaluation because the heuristic cannot label it reliably.

Context policy:

- `contextual` may use ancestor classes, ancestor resource IDs, and ancestor content descriptors.
- `local_only` must not read `ancestors` except to accept the argument; ancestor-only rules must not fire.

Canonical examples:

- Currency or price text in checkout/cart/payment/order/price context.
- Resource IDs containing `price`, `amount`, `total`, `account`, `uid`, `balance`, `transaction`.
- Account credentials, email, username, password, phone fields.
- Email-like text.
- CAPTCHA elements.
- Account/profile/order/transaction content descriptions.

Translatable examples:

- Header, title, section, label, and heading text.
- Buttons or labels in navigation/menu/tab contexts in `contextual` mode.
- TextView leaf nodes at moderate depth when not canonical.

Open examples:

- Non-critical images.
- Banners, promotions, ads, recommendations, hero/carousel regions.
- Remaining labeled nodes after higher-priority rules.

Implementation requirements:

- Use ordered rules; first match wins.
- Log label distribution by screen and aggregate split.
- Unit tests must explicitly show that an ancestor-only canonical rule fires in `contextual` mode and does not fire in `local_only` mode.

### 1.5 Gold Labels (`src/data/gold.py`)

Gold annotations are the authoritative final test labels.

Directory layout:

```
data/gold/
|-- raw_annotations/            # One file per annotator or batch
|-- overlap/                    # 10-screen overlap subset annotations
|-- disagreements/              # Generated disagreement reports
`-- gold_test_labels.csv        # Final resolved labels used for evaluation
```

Required annotation schema, CSV or JSONL:

| Field | Required | Description |
|-------|----------|-------------|
| `screen_id` | yes | Rico screen identifier |
| `node_id` | yes | Stable per-screen node ID from `flatten_hierarchy` |
| `label` | yes | `canonical`, `translatable`, `open`, or integer `0/1/2` |
| `annotator_id` | yes | Person or process that produced the label |
| `notes` | no | Free-form rationale or ambiguity note |

Gold workflow:

1. Select approximately 50 screens from apps that are excluded from heuristic train/val.
2. Include a 10-screen overlap subset labeled by at least two annotators.
3. Compute Cohen's kappa on the overlap subset.
4. Write disagreement reports to `data/gold/disagreements/`.
5. Resolve disagreements manually into `data/gold/gold_test_labels.csv`.
6. Validate that each `(screen_id, node_id)` exists in the corresponding flattened Rico graph.
7. Merge resolved gold labels into graph objects only for final evaluation.

Public interfaces:

```python

    """Write rows where annotators disagree."""
```

### 1.6 Graph Builder (`src/data/graph_builder.py`)

Public interface:

```python
def build_graph(
    flattened: FlattenedHierarchy,
    features: torch.Tensor,
    labels: list[int],
    include_sibling_edges: bool = True,
) -> dict:
    """
    Returns:
      {
        "x": FloatTensor [N, D],
        "y": LongTensor [N],
        "edge_index": LongTensor [2, E],
        "containment_edge_index": LongTensor [2, E_c],
        "sibling_edge_index": LongTensor [2, E_s],
        "node_ids": list[str],
        "parent_index": list[int | None],
        "ancestor_indices": list[list[int]],
        "feature_slices": dict[str, tuple[int, int]],
        "num_nodes": int,
      }
    """
```

Edge construction requirements:

- `flatten_hierarchy` stores raw containment edges as directed parent-child pairs.
- `build_graph` converts containment edges to bidirectional `edge_index` entries.
- `flatten_hierarchy` stores sibling edges as unordered pairs; `build_graph` converts each sibling pair to both directions.
- `include_sibling_edges=False` omits sibling edges from the combined `edge_index` but still stores `sibling_edge_index`.
- Do not add self-loops in `build_graph`. GCN/GAT layers add self-loops internally.

### 2.2 GCN (`src/models/gcn.py`)

Requirements:

- `build_graph` does not add self-loops.
- Every GCN layer adds self-loops internally.
- Scatter aggregation is performed in feature space before applying the layer weight.
- Aggregation tensor shape is `[N, in_dim]`, not `[N, out_dim]`.

```python
class GCNLayer(nn.Module):
    def __init__(self, in_dim: int, out_dim: int): ...

    def forward(self, x: torch.Tensor, edge_index: torch.LongTensor) -> torch.Tensor:
        """
        1. Add self-loops.
        2. Let edge_index[0] be source and edge_index[1] be destination.
        3. Mean-aggregate source features into each destination:
           agg = zeros([N, in_dim]); agg[dst] += x[src]; agg /= degree(dst)
        4. Apply linear transform: out = agg @ W.T + b.
        """
```

```python
class GCN(nn.Module):
    def __init__(
        self,
        in_dim: int,
        hidden_dim: int,
        num_classes: int = 3,
        num_layers: int = 2,
        dropout: float = 0.3,
    ): ...

    def forward(self, x: torch.Tensor, edge_index: torch.LongTensor) -> torch.Tensor:
        """Return [N, num_classes]."""
```

### 2.3 GAT (`src/models/gat.py`)

Requirements:

- Implement in PyTorch without PyG.
- Add self-loops inside each GAT layer.
- Use numerically stable destination-wise softmax over incoming edges.
- Final layer produces logits with no activation.

```python
class GATLayer(nn.Module):
    def __init__(self, in_dim: int, out_dim: int, num_heads: int = 4, dropout: float = 0.3): ...

    def forward(self, x: torch.Tensor, edge_index: torch.LongTensor) -> torch.Tensor:
        """Return [N, out_dim]."""
```

## Test Plan

### Unit Tests

Hierarchy flattening:

- Correct node count.
- Stable `node_id` assignment.
- Correct `parent_index`.
- Correct `ancestor_indices`.
- Correct depth, child count, and sibling count.
- Raw containment edges are parent-child pairs.
- Raw sibling edges are unordered pairs.

Graph building:

- Containment edges become bidirectional in `edge_index`.
- Sibling edges become bidirectional in `edge_index`.
- `include_sibling_edges=False` removes sibling edges from combined `edge_index`.
- Self-loops are absent from graph-builder output.
- `edge_index` dtype is `torch.long`.

Labeler:

- Currency in checkout/cart context is canonical in `contextual` mode.
- Same currency in promotional context is open or non-canonical when the rubric allows.
- Ancestor-only rules do not fire in `local_only` mode.
- Missing fields do not crash labeling.
- Unlabelable nodes return `None` from `assign_label` and `-1` from `label_graph`.

Gold utilities:

- Annotation schema validation catches missing required fields.
- Cohen's kappa is computed on the overlap subset.
- Disagreement report includes only conflicting labels.
- `attach_gold_labels` aligns by `(screen_id, node_id)`.

Feature extraction:

- Feature vector dimensions and `feature_slices` are correct.
- Empty text/content-desc gives a zero text embedding.
- Bounds normalize by actual screen size when available.
- Feature ablations preserve expected dimensions using saved slices.

Models:

- MLP, GCN, and GAT return `[num_nodes, 3]`.
- GCN and GAT self-loops work for isolated nodes.
- Batched disconnected graphs do not exchange messages.
- GCN aggregation uses `[N, in_dim]` before the linear transform.

## Key Design Decisions

| Decision | Choice | Reason |
|----------|--------|--------|
| Final test target | Hand-labeled gold labels | Avoids evaluating on the same heuristics used for supervision |
| Heuristic role | Train/validation only | Enables scale without making heuristic accuracy the final claim |
| Split granularity | App-level | Prevents leakage from repeated app layouts |
| Gold-app handling | Exclude from heuristic train/val | Prevents app-specific leakage into final test |
| Primary metric | Gold Macro-F1 | Weights rare canonical nodes equally with common open nodes |
| Headline comparison | GCN vs MLP on gold labels | Directly tests whether graph context helps |
| Circularity safeguard | `local_only` label-mode sanity check | Tests whether gains persist without ancestor-derived supervision |
| GCN aggregation | Mean aggregation | Simple and interpretable baseline |
| Self-loops | Inside model layers | Keeps graph construction model-agnostic |
| Feature ablations | Saved feature slices | Avoids recomputing graphs for every ablation |

## Common Pitfalls to Avoid

1. **Using heuristic labels as the final test set:** This invalidates the main claim. Gold labels are the final test set.
2. **Letting gold apps into heuristic train/val:** This leaks app-specific structure into final evaluation.
3. **Tuning on gold labels:** Gold labels are used after training, not during model selection.
4. **Ancestor circularity:** Report the local-only sanity check so graph gains are not only inherited from contextual heuristic rules.
5. **Screen-level splitting:** Always split by app.
6. **Wrong GCN scatter shape:** Aggregate into `[N, in_dim]`, then apply the layer weight.
7. **Implicit edge direction:** Store raw edges plainly and make bidirectionality explicit in `build_graph`.
8. **Feature ablation by recomputation:** Use `feature_slices`; recomputing risks inconsistent preprocessing.
9. **Embedding bottleneck:** Cache normalized text embeddings and make cache paths configurable.
10. **Loading all graphs at once:** Use lazy datasets and batched disconnected graphs.
